# Gos 进程派生（fork）与写时复制（COW）机制 —— 需求书

> 版本：v0.1（草案）　日期：2026-06-25　适用：Gos（generic operating system, RISC-V）

## 一、背景

Gos 是一套用于 RISC-V CPU/SOC 系统测试的软件程序，核心定位是**让开发者快速开发针对 RISC-V 架构/实现的测试用例，并尽可能多地暴露被测目标（硅前仿真 / FPGA / 流片）的问题**。测试用例以 command 形式存在（`app/command/`、`myUser/command/`），开发者通过 Shell 的 `command` / `user_run` / `vcpu_run` 驱动执行。

当前 Gos 已具备较完整的 MMU/页表、缺页入口、每任务独立页表（`struct task.pgdp` + 带 ASID 的 satp）、TLB 精确刷新（`sfence.vma` / `sinval`）、原子与多核测试等能力，但**没有进程派生（fork）语义，也没有写时复制（COW）机制**。本需求书论证为 Gos 引入「轻量 fork + COW」的必要性，并界定需求范围。

## 二、问题陈述：现有测试能力的缺口

经代码梳理，现状与缺口如下：

| 被测能力 | 现有覆盖 | 缺口 |
|---|---|---|
| MMU / 页表 / A·D 位 | `app/command/page_tlb_test.c` | 仅单地址空间、静态权限 |
| TLB / `sfence.vma` / `sinval` / ASID | `page_tlb_test.c`、`sinval_test.c` | 无「权限翻转→缺页→修复→重试」的闭环场景 |
| 原子指令 AMO / LR-SC / CAS | `spinlock_test.c`、`multicore/cycas.c` | 未在「跨地址空间共享页 + 写保护」下验证 |
| 多 hart / 共享内存 | `multicore_memory_test/` | 共享页在同一地址空间内，无跨地址空间 COW 竞争 |
| 缺页异常处理 | `mm/pgtable.c:do_page_fault()` 仅做 VMAP 惰性分配 | **PTE 有效即返回 -1，写保护缺页根本走不到**（`mm/pgtable.c:682`） |
| 多地址空间隔离 / 上下文切换 | `create_user_task()` 存在 | 仅 `memcpy` 顶层 PGD，无深拷贝、无系统性隔离测试 |
| 写时复制 / 共享只读页 | 无 | **完全缺失** |

**核心矛盾**：Gos 想测的最难、最易出 bug 的子系统恰恰是 MMU/TLB/原子/多核协同，而这些子系统中「写保护缺页投递 → 软件修复 → 重新执行」这条最关键的硬件路径，目前 Gos 没有任何测试能够触发。COW 正是把这条路径作为**正常控制流**反复、密集触发的机制。

## 三、价值论证：fork + COW 为什么是「测试放大器」

引入 fork+COW 有两层价值。

### 3.1 对测试开发者：降低编写隔离/并发用例的成本

- 现状要构造「两个独立地址空间、共享某些页」需手工 `create_user_task` + 逐页 `mmu_user_page_mapping`，样板代码多、易错。
- 提供 `fork()` 后，开发者一行调用即可得到一个与父几乎一致、彼此隔离的地址空间，自然地表达「父子并发写同一逻辑页」「隔离性验证」等经典用例，测试代码更短、更贴近意图。

### 3.2 对被测 RISC-V 实现：作为高强度工作负载暴露硬件问题（主价值）

COW 不是普通功能，而是一个**针对 MMU/TLB/原子/多核的确定性高压工作负载**，可系统性触发以下易出 bug 的硬件行为：

1. **Store/AMO 写保护缺页投递（scause=15）**：在「有效但只读」的页上写，要求精确异常、`stval/sepc` 正确、修复后可正确重执行。这是 MMU 实现最易出错处之一，当前 Gos 从不触发。
2. **硬件 A/D 位更新（Svadu）对比陷入式更新**：COW 清写位后再写，考验 D 位置位 / 触发陷入的状态机是否正确。
3. **TLB 一致性与 `sfence.vma` 语义**：COW 改 PTE（清写位、装新页）后必须正确失效旧 TLB 项；多 hart 下需 IPI shootdown。这是典型的硅 bug 来源。
4. **ASID 标记的 TLB 正确性**：fork 产生新地址空间 / 新 ASID，父子共享同一物理页但 ASID 不同，考验 ASID 作用域的 `sfence.vma`/`sinval`。
5. **共享/COW 页上的原子操作**：LR/SC 预约、AMO 与缺页/TLB 重填的交互——预约是否在写保护陷入后失效、顺序是否正确。
6. **多 hart 竞争同一 COW 页**：两核同时在同一只读共享页缺页，考验内存序、预约、并发 PTW、引用计数的原子性。
7. **超页与 COW 的交互**（可选增强）：若共享区含 2M/1G 大页，COW 触发的超页拆分考验 PTW 与拆分逻辑。

> 一句话：现有 `page_tlb_test` / `sinval_test` / `cycas` 各测一个点，COW 把它们**编织成一条端到端、可自检、可多核并发**的链路，覆盖单点测试覆盖不到的组合态。

## 四、需求目标

为 Gos 增加一套**轻量、面向测试**的 fork + COW 机制，使开发者能以最小代价构造上述测试场景，并使被测实现的 MMU/TLB/原子/多核路径得到高强度、可自检的验证。

## 五、需求范围（Scope）

### 5.1 功能性需求（FR）

- **FR-1 物理页引用计数**：为物理页帧提供引用计数能力，支持多映射共享同一物理页，归零才真正释放。
- **FR-2 PTE COW 标记与权限原语**：定义 COW 软件标记位（复用 `_PAGE_SOFT`），提供「写保护 / 标记 COW / 判定 COW / 统一写 PTE」原语。
- **FR-3 地址空间复制（copy_page_range）**：遍历父用户地址空间叶子 PTE，将父子两侧置为「共享 + 只读 + COW」，对共享物理页增引用计数。
- **FR-4 fork 接口**：提供内核接口与 U 态系统调用（`sys_fork`），创建子任务并复制地址空间；父返回子 id、子返回 0（语义可裁剪，见 §六）。
- **FR-5 COW 缺页处理**：扩展 `do_page_fault`，识别写保护缺页：引用计数为 1 时就地恢复写权限；大于 1 时分配新页、拷贝、换 PTE、对旧页减引用、精确刷 TLB，并令故障指令可重执行。
- **FR-6 故障原因传递**：缺页路径需将 `scause`（区分 load/store/instruction）与发生特权级传入处理函数。
- **FR-7 测试 command**：提供至少一组 COW/fork 自检测试（核心态与 U 态各一），可作为开发者编写测试的模板。

### 5.2 非功能性需求（NFR）

- **NFR-1 可观测**：缺页/拷贝/引用计数变化可打印（受调试开关控制），便于定位被测硬件问题。
- **NFR-2 多核安全**：引用计数与 PTE 修改在多 hart 下并发安全（原子操作 / 锁）。
- **NFR-3 可裁剪**：通过 Kconfig 开关控制是否编译 COW/fork，不影响既有配置。
- **NFR-4 风格一致**：复用现有 `mm`、`task`、`user`、`syscall`、`command` 模块的既有接口与代码风格（参考 Linux/Xvisor 的既有约定）。
- **NFR-5 自检性**：测试用例自带期望值校验（pass/fail 明确输出），适配 FPGA/VCS/帕拉丁等无人值守环境。

### 5.3 不在本期范围（Out of Scope）

- 完整 POSIX fork/exec/wait 语义、信号、文件描述符继承。
- 用户态 `malloc`/堆与 COW 的深度整合。
- 换页到外存（swap）、内存回收（reclaim）、反向映射（rmap）通用框架——本期只做支撑 COW 测试所需的最小集。

## 六、关键设计取舍（需评审确认）

1. **fork 语义裁剪**：Gos 是测试 OS，不必照搬 Unix fork。建议实现「派生一个共享父页（COW）的隔离子地址空间」即可，不强求复制全部内核态资源。
2. **COW 范围**：默认仅对用户区可写匿名页做 COW；共享内存区（`user_share_memory_*`）、设备映射不纳入。
3. **引用计数粒度**：建议按 PFN 建数组（覆盖物理内存范围），实现简单、查询 O(1)；与现有 bitmap 分配器（`struct mem_maps`，`include/gos/mm.h:28`）解耦。
4. **是否支持超页 COW**：本期可先要求 COW 区按 4K 粒度映射，超页拆分作为后续增强。

## 七、验收标准（高层）

- **AC-1**：U 态测试中父 `fork` 出子，父子各自写同一逻辑地址，互不可见对方修改（隔离性）。
- **AC-2**：缺页计数与引用计数符合预期——只读共享页被写时触发恰好一次写保护缺页并完成拷贝；最后一个引用写时不再拷贝。
- **AC-3**：多 hart 并发对同一 COW 页写入，结果自洽、无丢失/串扰，TLB 无陈旧项导致的错误。
- **AC-4**：关闭 COW 的 Kconfig 配置下，既有全部 command 测试与编译不受影响。
- **AC-5**：在 QEMU 与至少一种目标平台（FPGA/VCS 其一）上可运行并给出明确 pass/fail。

## 八、风险

- **R-1 缺页处理改动影响面**：`do_page_fault` 现有 VMAP 惰性分配逻辑不可回归（`mm/pgtable.c:672`）。
- **R-2 多核竞争**：引用计数/PTE 修改的并发正确性是难点，也是最易暴露被测实现内存序问题之处（既是风险也是价值）。
- **R-3 超页/NAPOT 交互**：与 `svnapot`、2M/1G 映射并存时的拆分复杂度。
- **R-4 ASID 复用**：fork 分配新 ASID 与现有 ASID 管理需对齐，避免 TLB 串扰被误判为硬件 bug。
